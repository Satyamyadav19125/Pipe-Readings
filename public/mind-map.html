💧 PVC Pipe Readings — how the tool works

Digital Village Project · Kharif 26 · AWD paddy irrigation monitoring across Punjab villages

Tap any branch to expand it — the tree goes from the field surveyor all the way down to what you see on screen.

1 · Where the data is born — the field
A surveyor stands at a PVC observation pipe in a paddy field and opens the Kobo form on their phone.
📋 The Kobo form: "Kharif 26 – PVC Pipes"
Hosted on kf.kobotoolbox.org, asset UID aytG3bjuzKg92S3QihTJ6J. Questions the surveyor fills:
  • Date & start time — when the visit happened
  • Name — which surveyor (must match the Team list)
  • Village → Farm → Pipe — cascading choice lists; ~1,188 pipes across all villages
  • Outside height (mm) — pipe measured from ground to top; standard 150 mm
  • Water level inside (mm) — tape lowered into the pipe; valid 50–250 mm
  • Photo of tape inside + field photo — two pictures per visit
  • GPS location — captured automatically at the pipe
⬇️
2 · How data reaches the dashboard
⚡ Webhook (instant path)
Kobo's REST Services is registered with the URL https://pipe-readings-lac.vercel.app/api/webhook?secret=…. Every time a form is submitted, Kobo POSTs there, and the dashboard clears its cache so the new reading appears within seconds. If this URL points at an old/deleted deployment, Kobo emails a REST Services Failure Report — the data is safe, only the instant ping fails.
⏱️ 30-second cache (normal path)
Pages fetch submissions from the Kobo API (/api/v2/assets/…/data/) using the API token, through Next.js's Data Cache with a 30-second revalidation. So even without the webhook, the dashboard is never more than ~30 s behind.
↻ Manual Refresh button
The header's Refresh calls /api/refresh, which busts the cache immediately — same effect as the webhook, triggered by hand.
🧠 The form definition (master list)
Besides submissions, the tool also downloads the form's choice lists (every village, farm and pipe that exists). That's how it knows a pipe with zero submissions still exists — powering "95 / 1188 done" and the per-village to-take counts.
⬇️
3 · Where things are stored
☁️ KoboToolbox (source of truth)
All readings and photos live on Kobo's servers. The dashboard never copies them into its own database — it reads them live (cached 30 s). Deleting the dashboard loses nothing.
🍃 MongoDB Atlas — database pipe_readings
Only the tool's own data lives here, isolated from the Water Meter app:
  • settings — project info, red-flag toggles, pipe standards, Kobo forms, contact people, admin passwords
  • assignments — surveyors, their villages, passwords, photos
  • messages + chat_reads — team chat and unread tracking
  • tasks, media, verified — to-dos, uploaded chat files, "mark as correct" flags
▲ Vercel (the app itself)
The Next.js app deployed from the GitHub repo. Every commit to main auto-deploys in ~1 minute. Environment variables hold the Mongo URI, Kobo token/UID, admin passwords fallback and webhook secret.
⬇️
4 · The brain — what happens to every reading
🗺️ Field map (3-pass matcher)
lib/fieldMap.js translates Kobo's raw keys (group_2/Readings_mm) into logical names (reading, village, pipe, photo, GPS). It matches exact → case-insensitive → last-segment, so renaming a group in Kobo doesn't break the dashboard.
🚩 Red-flag engine
Every submission is checked against the rules in lib/redflags.js:
  • Missing photo — no picture attached
  • Stale 7 days — a pipe's latest reading is older than a week
  • Inside out of range — water level outside the 50–250 mm standard (Settings → Pipe parameters)
  • Outside differs from standard — pipe height ≠ 150 mm (± tolerance) → sunk / lifted / wrong measurement
  • Stuck readings, future dates, out-of-sequence dates, plus opt-in GPS/duplicate/speed checks
Admins can toggle every rule and edit the standards in Settings; surveyors never see flags.
📈 Weekly targets
lib/weekly.js groups readings per pipe per period (default: 1 reading / week) and classifies each pipe as done, in progress or needs reading — merged with the master list so never-read pipes count too.
⬇️
5 · What you see — every tab explained
🏠 Overview
KPIs: total submissions, this-week progress out of all 1,188 pipes, villages, surveyors, flags. Surveyors see only their own villages.
📋 Submissions
Every reading as a card — water level (mm), both photos labeled, flag details, "mark as correct" for admins, filters by village/surveyor/date/flag.
💧 Water level
Per-pipe pairs of consecutive readings showing how the level moved between visits (rise and fall are both normal in AWD — never flagged).
🗺️ Map
Every GPS-tagged reading as a pin — blue clean, red flagged. Street/Satellite/Topo layers with automatic fallback if street tiles fail, heat-map mode, pin popups with the full submission info, KML/GeoJSON/HTML export for your own cartography.
👥 Assignment (3 sub-tabs)
Team — people, passwords, photos and village assignments. Tasks — to-dos. Assignment — done / left / total readings per village, collapsible per village, for this week or any past week.
💬 Chat
Group + per-surveyor DMs. Photos (camera or gallery), voice notes, documents, location and live location. Unread badges on the tab and per room, browser notifications on new messages.
🪞 Kobo View (admin)
Spreadsheet of raw submissions — every answered question, both photos, mini map with the pin, deep link to open that exact submission on the real KoboToolbox site.
⚙️ Settings (admin)
Kobo forms (active UID shown), project info, reading targets, pipe standards, photo quality, contact people for the landing page, red-flag toggles, admin passwords, data & storage with XLSX/CSV/flag/map downloads per month.
⬇️
6 · Getting data back out
📗 Excel (XLSX)
Two sheets: Readings (all rows) and Summary — totals, averages, min/max water level, per-village readings/pipes/last-reading table.
📊 CSV / JSON
Same rows for scripts, R, Python or Google Sheets.
🗺️ Map files
KML (Google Earth), GeoJSON (QGIS/ArcGIS) and a standalone HTML map — ready for the project's cartographic outputs.
7 · Who can see what
Admins (passwords in Settings → Admin passwords, or the ADMIN_PASSWORD env var) see everything.
Surveyors (password set per person in Assignment → Team) see only their villages, no flags, no Kobo View, no Settings — a clean, positive field view.
